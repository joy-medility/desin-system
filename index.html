PillEye Design System · v1.0

정확한 카운팅, 흔들림 없는 신뢰.

약사를 위한 비전 AI 기반의 PillEye B2B 환경에 맞는 다크모드 기반으로 신뢰감을 전달합니다. 절제된 색상 대비와 명확한 타이포그래피 위계, 즉각적인 상태 피드백을 통해 신뢰성을 화면 곳곳에서 시각적으로 구현해 더 나은 사용자 경험을 제공합니다.

Core Value

1. Trusted Accuracy
오류가 없는 곳에서, 신뢰가 시작됩니다.

Enterprise-grade counting accuracy with built-in error detection.

엔터프라이즈 수준의 정확도와 오류 감지 기능을 제공합니다.
단순한 카운팅을 넘어, 복잡한 제형 인식과 wrong drug flagging까지 실시간으로 처리합니다. 사람이 놓칠 수 있는 오류를 시스템이 먼저 잡아내어, 약사가 최종 판단에만 집중할 수 있는 환경을 만듭니다.

2. Real-Time Workflow Automation
설치도, 적응할 시간도, 기다림도 필요하지 않습니다.

Instant deployment with seamless workflow integration.

즉시 배포 가능하며 기존 워크플로우에 자연스럽게 통합됩니다.
인력 부족 상황에서도 즉시 투입 가능한 Zero-install 구조로, 기존 시스템을 바꾸지 않고 그대로 연결됩니다. 현장이 기술에 맞춰야 하는 부담 없이, 도입 첫날부터 운영 효율이 올라갑니다.

3. Scalable Cost Efficiency
초기 투자 비용 없이, 바로 사용할 수 있습니다.

Subscription-based model designed for enterprise rollout.

엔터프라이즈 확장을 고려한 구독 기반 모델을 제공합니다.
하드웨어 구매 없이 구독 방식으로 시작하여 CapEx를 제거하고, 체인 약국·병원 그룹 단위의 대규모 확장도 동일한 구조로 대응합니다. 규모가 커질수록 효율도 함께 커집니다.

Foundation / Type

Text

일관된 타이포그래피 시스템을 제공하여 다양한 UI 구성 요소에 적용할 수 있도록 합니다.

Display

화면에서 가장 크고 강조되어야 하는 요소에 사용합니다. 주로 카운팅 숫자, 핵심 수치처럼 한눈에 들어와야 하는 단일 정보에 적합합니다. 숫자로 쓰일 때는 3% 자간을, 텍스트로 쓰일 때는 0% 자간을 적용합니다.

Heading

화면 내 섹션을 구분하거나 팝업/모달의 제목에 사용합니다. Display보다 작지만 Body보다 크고 굵어서 위계를 구분해줍니다.

Body

UI에서 가장 많이 쓰이는 텍스트 유형입니다. 버튼 레이블, 설명 텍스트, 상태 메시지처럼 읽어야 하는 텍스트에 사용합니다.

Caption

화면에서 가장 작은 텍스트입니다. 주요 정보를 보조하는 역할입니다. 단독으로 쓰기보다 다른 텍스트와 함께 사용합니다.

Don't

  • 한 화면에 display는 1~2개 이내로 제한합니다. 많을수록 강조 효과가 희석됩니다.
  • Display는 긴 문장에 사용하지 않습니다. (20글자 이하로 사용)
  • 숫자 단독 표시 시 tracked 변형(자간 3%)을 사용해요. 텍스트 혼용 시엔 기본(0%)을 사용합니다.
  • Caption은 항상 보조 역할입니다. 사용자가 반드시 읽어야 하는 정보에는 쓰지 않아요. Ex) Rx 번호나 NDC 같은 핵심 식별 정보는 body 계열로 올리고, Caption은 부가 정보에만 사용
  • 버튼 레이블에 Caption을 사용하지 않습니다. Caption(14px)은 터치 영역에서 가독성이 떨어지기 때문에 버튼은 최소 Body-3(18px) 이상을 사용해야 합니다.
Foundation / Shape

Radius

XL~XS 상대 크기 체계로 반지름 값을 변환한 토큰입니다. 시각적 중요도·형태 비율·레이아웃 맥락에 맞춰 일관된 Radius를 적용합니다.

Don't

  • 같은 위계의 버튼에 다른 Radius를 쓰지 않습니다. Ex) Cancel Fill, Partial Fill, Complete Fill은 같은 위계의 버튼이므로 동일한 Radius 값을 사용합니다.
  • 작은 컴포넌트에 큰 Radius를 쓰지 않습니다. Ex) 작은 칩/태그 컴포넌트는 corner-radius-XS(4px) 또는 corner-radius-S(6px)를 사용하고 circle은 완전한 원형이 필요할 때만 사용합니다.
  • 모달/다이얼로그에 작은 Radius를 쓰지 않으며 corner-radius-L(12px) 이상부터 적용해서 사용합니다.
  • 중첩된 박스에 같은 Radius를 쓰지 않습니다. 안쪽 박스는 바깥 박스보다 한 단계 작은 Radius를 사용합니다. Ex) 바깥 XL(16px) → 안쪽 M(8px) 또는 S(6px).
  • Radius를 0으로 쓰지 않습니다. 최소 corner-radius-XS(4px) 이상을 사용해요. Radius 0은 토큰에 정의되지 않은 값입니다.
  • 토큰에 없는 임의의 Radius 값을 쓰지 않습니다. 반드시 정의된 토큰 중에서 선택합니다.

오버레이 전역 규칙 (모든 팝업 공통 — Figma에 명시됨)

타입과 무관하게 항상 적용됩니다.

  • Dim은 항상 전체 화면 — 팝업 위치(중앙/특정 버튼 근처)와 무관하게 배경은 항상 전체 dim 처리
  • 팝업이 열리면 하단 버튼은 항상 비활성화 — 실수 터치를 원천 차단
  • 영역 외 터치(tap outside) 허용 여부는 타입별로 다름 — 이탈 컨펌류는 영역외터치 허용(X와 동일). 파괴적 액션은 허용 여부 불명 → 미확정
  • 팝업 노출 중 카메라/카운트 프리징 + 수식 표시 숨김 — 배경 실시간 카운트 수식 표시를 숨김 (정보 과부하 방지)
팝업 타입 분류 (실제 사례 기반 — 5개)

기존 5개 분류(확인/경고/오류/정보/진행)가 실제로도 거의 맞지만, 이탈 확인은 화면마다 패턴이 달라 별도 서브타입으로 둡니다.

타입실제 사례
확인형 — 이탈 컨펌Leave Simple Count?, Rx Count 이탈 시 저장 여부 확인
경고형 — 파괴적 액션 컨펌Cancel this fill?, Delete (running total), Reset→Zero-out
경고형 — 비즈니스 룰 예외Flagged item 존재 시 Add to Running Total → Ignore or Try Again
오류형 (Error/Alert)+ 버튼 누를 때 오류 시 알림 팝업
정보형 — 토스트작업 완료/상태 알림 (노란색 vs 화이트 컬러 미정)

→ 진행형(Progress)은 이번 검색 결과에선 나오지 않았어요 — 카운팅 중 로딩 UI 별도 여부는 세션에서 확인이 필요합니다.

1. 확인형 — 이탈 컨펌 (Leave / Exit Confirm)

화면마다 패턴이 달라 화면별 매트릭스로 정리했어요 — 단일 템플릿으로 통일할 수 없어요.

화면타이틀본문버튼 구성비고
Simple Count"Leave Simple Count?""All progress will be lost. This cannot be undone"Leave(Red, 진행) / Stay(Grey, Cancel 역할)X 버튼 필요, 영역외터치 허용
Rx Count"Would you like to save your Rx progress before leaving?"—Save and Leave(Green) / Stay(Grey, Cancel 역할) / Leave without Saving(Red 텍스트, 'Cancel fill'과 동일 동작)X 버튼 필요, 영역외터치 허용. 3버튼 구성
Rx Count (Save and Leave 후 복귀)Resume to Recount(Rx Count 팝업과 동일 노출 & flow)동일단, Cancel은 팝업닫힘 액션. 타 메뉴에서 햄버거 메뉴가 펼쳐진 상태로 진입
My Record———No message (컨펌 없이 바로 이탈)
Settings———No message

관찰: 작업 손실(progress loss) 여부가 컨펌 유무를 가릅니다. My Record/Settings는 손실될 게 없어 컨펌을 생략 — "왜 이 화면엔 팝업이 있고 저 화면엔 없는지" 판단 기준이 돼요.

2. 경고형 — 파괴적 액션 컨펌
Cancel Fill
  • 트리거: Cancel Fill 버튼 (Review & Capture 화면의 Cancel 버튼도 동일)
  • 타이틀: "Cancel this fill?"
  • 본문: "Confirm Cancel to permanently delete all data for [Rx #1234567]. This will not be recorded" — Rx 번호는 bold 변수 삽입
  • 비고: Review & Capture에서는 본문 중 "Return to Counting"이 "Return to Review"로만 교체 (구조 재사용)
Delete (Running Total에서 항목 삭제)
  • 위치: 프리뷰 센터
  • 트리거 중: Counting 및 카메라 프리징, Reset, Add to Running Total 비활성화
  • 최신 버전(Verified Count 도입 후): delete 전 최종 running total을 먼저 보여주고 subtract될 것을 알림 — "몇 개가 얼마로 줄어드는지"를 먼저 보여주는 패턴
Reset → Zero-out (명칭 변경됨)
  • 형태: 말풍선(speech bubble), Reset 버튼 근처 (중앙 모달 아님)
  • 동작: Counting은 Total만 노출, 카메라 프리징, Add to Running Total 비활성화

→ 공통 패턴: 파괴적 액션 직전에 구체적 손실을 명시합니다 (전역 톤 원칙 "구체적으로"와 연동).

3. 경고형 — 비즈니스 룰 예외
Flagged item 존재 시 Add to Running Total
  • 형태: 말풍선, Add to Running Total 버튼 근처
  • 동작: Counting 및 카메라 프리징
  • 버튼: Ignore / Try Again
  • Ignore 선택 시 → 정상적으로 다음 단계 진행 (구체적 다음 동작은 세션에서 확인)
4. 오류형 (Error / Alert)

트리거: 오류 발생 시 + 버튼을 누르면 알림 팝업 노출. 세부 카피/레이아웃은 미확정 — 세션에서 채울 것 (인라인 에러 §3와 역할 분담 포함).

추가로 발견된 룰 (분류 안 됨 — 세션에서 타입 배정)
  • 이미지 클릭 시 UI 변경 + 팝업 노출: 이미지 확대/프리뷰 추정 — 별도 타입 vs 정보형 변형 확인 필요
  • 삭제 컨펌(tbd): OK 누르면 합계에서 빠짐 — 위 Delete와 동일 패턴인지, 리스트 아이템 단위 삭제인지 구분 필요
  • Unit of Use 버튼 재입력: 입력 후 동일 버튼 재탭 시 동일 팝업 재노출 (멱등성)
  • 이탈 컨펌 매트릭스 보완: Workflow 전체, Login 등 빠진 화면 확인

세션 우선 논의 사항

  1. 이탈 컨펌 매트릭스 완성 — §4 표에 빠진 화면 채우기
  2. "구체적 손실 보여주기" — 모든 파괴적 액션에 일관 적용 여부 (Delete는 적용, Cancel Fill은 아직 추상적)
  3. 토스트 컬러 (노랑 vs 화이트)
  4. 영역외터치 허용/비허용 — 타입별 표
  5. 이미지 클릭→팝업 — 정식 타입 vs 예외
  6. §1~3 미확정 항목 — 말투, 버튼, 에러/빈 상태/로딩, 약학 용어 일괄 확정
  7. 용어 통일 — Cancel vs Leave vs Delete vs Zero-out (Reset→Zero-out 변경 사례 참고)
